# セキュリティ準拠設計（医療・介護ガイドライン対応）

## 目的と範囲
- 国内リージョン前提で、医療・介護の安全管理ガイドライン遵守を満たす設計を提示する
- 本ドキュメントは「技術設計 + 運用設計」の骨子を定義する

## 適用する基準（方針）
- 医療情報システムの安全管理ガイドライン（厚労省）
- 医療・介護関係事業者の個人情報取扱いガイダンス（厚労省 / 個人情報保護委員会）
- 医療情報を取り扱う情報システム提供事業者の安全管理ガイドライン（経産省/総務省）

## セキュリティ要件（要約）
### 1. ガバナンス / 組織
- 情報管理規程（責任分界、SLA、委託管理）
- 事故対応手順（報告・連絡・封じ込め・再発防止）
- BCP / 復旧手順（RTO/RPO の明文化）

### 2. アクセス制御
- 最小権限（IAM ロール分離）
- 管理者操作の監査
- テナント分離（`tenantId` 強制）

### 3. データ保護
- 通信暗号化（TLS 強制）
- 保存時暗号化（GCP 標準暗号化、必要時は KMS）
- 重要情報の秘匿（Secret Manager）

### 4. 監査・ログ
- 操作ログ・アクセスログの保持
- 管理者の監査証跡
- 監査ログの改ざん防止（保管ポリシー）

### 5. 可用性 / 復旧
- バックアップの取得と復元訓練
- 障害時の切り戻し手順

### 6. 個人情報保護
- 目的外利用の禁止
- 保存期間 / 削除方針の明文化
- 開示・訂正・削除への対応手順

## GCP 構成（国内リージョン / 準拠優先）
### 推奨構成
- API: Cloud Run（asia-northeast1）
- DB: Firestore（asia-northeast1）
- 認証: Google OAuth
- 監査ログ: Cloud Logging / Cloud Audit Logs
- Secrets: Secret Manager

### 課金注意（無料枠が事実上ない可能性が高い）
- Cloud SQL（代替として Firestore を採用）
- Secret Manager（小額でも課金対象になり得る）
- KMS（必要時のみ）
- Cloud Armor（WAF 相当）
- Firestore バックアップ / エクスポート

※ 上記は「無料枠が 0 の可能性が高い」ため、導入時は必ず課金確認すること。

## 実装設計（概要）
### 認証・認可
- Google OAuth による認証
- JWT を発行し、API では `tenantId` を必須チェック
- 管理系 API はロール制御

### 監査ログ
- 認証・操作・重要データ変更のログを記録
- ログの保存期間とアクセス権限を明確化

### データ暗号化
- 通信は TLS 1.2+ を必須
- Firestore の標準暗号化を前提
- 重要情報は Secret Manager へ保管

### バックアップ / 復旧
- Firestore バックアップ/エクスポートの周期運用
- RTO / RPO を決定し文書化

## 運用設計（必要最低限）
- 情報管理規程（ひな形を別途作成）
- 事故対応フロー（連絡先・エスカレーション）
- 監査ログの定期レビュー

## 次に作成する成果物
- 準拠要求 → 実装対応表（チェックリスト）
- 運用規程のひな形
- 監査ログ・バックアップ運用の詳細設計
