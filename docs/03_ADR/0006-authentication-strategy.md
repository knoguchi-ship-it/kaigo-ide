# ADR-0006: 認証方式の選定

**ステータス**: Accepted
**日付**: 2026-01-29
**決定者**: [アーキテクト名]

## コンテキスト

KaigoIDEの認証・認可方式を選定する必要がある。

要件：
- セキュアなユーザー認証
- セッション管理
- マルチデバイス対応（PC、タブレット、スマホ）
- オフライン対応（PWA）との整合性
- 将来的な2要素認証のサポート

## 決定

**JWT (JSON Web Token) による認証を採用する。**

構成：
- アクセストークン: 短寿命（15分）、メモリ保持
- リフレッシュトークン: 長寿命（7日）、HTTP Only Cookie

## 選択肢

### 選択肢1: JWT認証
- **概要**: ステートレスなトークンベース認証
- **メリット**:
  - ステートレスでスケーラブル
  - マイクロサービスに適している
  - モバイル/SPAとの相性が良い
  - オフライン時もトークンでユーザー情報参照可能
- **デメリット**:
  - トークン無効化が難しい（Blacklist必要）
  - トークンサイズがセッションIDより大きい

### 選択肢2: セッションベース認証
- **概要**: サーバーサイドでセッションを管理
- **メリット**:
  - シンプルで実績がある
  - 即座にセッション無効化可能
  - トークンサイズが小さい
- **デメリット**:
  - サーバー側で状態管理が必要
  - スケーリング時にセッション共有が必要
  - SPAとの相性がやや悪い

### 選択肢3: OAuth 2.0 / OpenID Connect
- **概要**: 外部認証プロバイダーを使用
- **メリット**:
  - ソーシャルログイン対応
  - 認証の責任を外部に委譲
  - 標準仕様
- **デメリット**:
  - 介護事業所ユーザーにはソーシャルログインの需要が低い
  - 外部サービスへの依存
  - 実装が複雑

## 決定理由

1. **SPA/PWA適合**: JWTはSPAやPWAでの認証に適している
2. **スケーラビリティ**: ステートレスなのでサーバーのスケーリングが容易
3. **オフライン対応**: トークン内のユーザー情報をオフライン時に参照可能
4. **NestJS統合**: PassportのJWT戦略が標準サポート
5. **セキュリティ**: リフレッシュトークンをHTTP Only Cookieで保護

## 詳細設計

### トークン構成

```
┌─────────────────────────────────────────────────┐
│ Access Token (JWT)                              │
├─────────────────────────────────────────────────┤
│ Header: { alg: "RS256", typ: "JWT" }            │
│ Payload: {                                      │
│   sub: "user_id",                               │
│   tenantId: "tenant_id",                        │
│   role: "USER",                                 │
│   iat: 1234567890,                              │
│   exp: 1234568790 (15分後)                      │
│ }                                               │
│ Signature: RS256署名                            │
└─────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────┐
│ Refresh Token                                   │
├─────────────────────────────────────────────────┤
│ - UUID形式                                      │
│ - DBに保存（ユーザーID、有効期限、デバイス情報）   │
│ - HTTP Only Cookie で送信                       │
│ - 有効期限: 7日                                 │
└─────────────────────────────────────────────────┘
```

### 認証フロー

```
1. ログイン
   POST /auth/login { email, password }
   ↓
   検証成功 → Access Token (Body) + Refresh Token (Cookie)

2. API呼び出し
   GET /api/... Authorization: Bearer {access_token}
   ↓
   JwtAuthGuard → JwtStrategy → 検証 → ユーザー情報注入

3. トークンリフレッシュ
   POST /auth/refresh (Cookie: refresh_token)
   ↓
   新しい Access Token + (ローテーション時) 新しい Refresh Token

4. ログアウト
   POST /auth/logout
   ↓
   Refresh Token を DB から削除 + Cookie クリア
```

### セキュリティ対策

| 脅威 | 対策 |
|------|------|
| XSS | Access TokenはメモリのみでlocalStorage不使用 |
| CSRF | Refresh TokenはHTTP Only Cookie + SameSite=Strict |
| トークン窃取 | 短いAccess Token有効期限（15分） |
| リプレイ攻撃 | Refresh Tokenローテーション |
| 不正利用 | Refresh Tokenのデバイス紐付け、異常検知 |

## 影響

### ポジティブな影響
- SPAでの認証がスムーズ
- サーバーのスケーリングが容易
- PWAのオフライン対応と整合
- 将来的な2FA追加が容易

### ネガティブな影響
- アクセストークン即時無効化には追加実装が必要
- リフレッシュトークン管理のDB負荷

### リスクと軽減策
| リスク | 軽減策 |
|--------|--------|
| トークン漏洩 | 短いアクセストークン有効期限、HTTPS必須 |
| セッションハイジャック | デバイス情報の検証、異常ログイン通知 |
| Refresh Token無限利用 | ローテーション、定期的な再認証要求 |
