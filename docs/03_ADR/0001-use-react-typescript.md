# ADR-0001: React + TypeScriptの採用

**ステータス**: Accepted
**日付**: 2026-01-29
**決定者**: [アーキテクト名]

## コンテキスト

KaigoIDEはブラウザベースのSPAとして開発する。フロントエンドフレームワークと言語を選定する必要がある。以下の要件を満たす必要がある：

- PC、タブレット、スマートフォンでのレスポンシブ対応
- PWA対応によるオフライン機能
- 複数の機能モジュールを独立して開発・保守できる構造
- 長期的な保守性と拡張性
- 開発者の採用しやすさ

## 決定

**React 18 + TypeScript 5 をフロントエンドの主要技術として採用する。**

ビルドツールにはViteを採用し、高速な開発体験を実現する。

## 選択肢

### 選択肢1: React + TypeScript
- **概要**: Metaが開発するUIライブラリ + 静的型付け言語
- **メリット**:
  - 最大のエコシステムとコミュニティ
  - 豊富なライブラリ・ツール
  - 型安全性による保守性向上
  - 採用市場が最も大きく人材確保しやすい
  - PWA対応が容易
- **デメリット**:
  - 設計の自由度が高い分、規約が必要
  - 状態管理など追加ライブラリの選定が必要

### 選択肢2: Vue.js + TypeScript
- **概要**: 学習コストが低いプログレッシブフレームワーク
- **メリット**:
  - 学習コストが低い
  - 公式のルーター・状態管理がある
  - 日本での人気が高い
- **デメリット**:
  - TypeScriptサポートはReactより後発
  - エコシステムがReactより小さい
  - 大規模開発の実績が少ない

### 選択肢3: Next.js (React + SSR)
- **概要**: Reactベースのフルスタックフレームワーク
- **メリット**:
  - SSR/SSGによるSEO・初期表示最適化
  - ファイルベースルーティング
  - API Routes内蔵
- **デメリット**:
  - 介護業務システムにSSRのメリットが薄い
  - フレームワークへの依存度が高い
  - バックエンドを別途開発する方針と合わない

### 選択肢4: Angular + TypeScript
- **概要**: Googleが開発するフルスタックフレームワーク
- **メリット**:
  - 規約が明確でチーム開発しやすい
  - TypeScriptがデフォルト
  - 企業システム開発での実績
- **デメリット**:
  - 学習コストが高い
  - バンドルサイズが大きくなりがち
  - 採用市場がReactより小さい

## 決定理由

1. **エコシステム**: Reactは最大のエコシステムを持ち、必要な機能のライブラリが揃っている
2. **TypeScript親和性**: React + TypeScriptの組み合わせは広く普及し、型定義も充実している
3. **PWA対応**: Service Worker、Workboxとの統合が容易
4. **採用市場**: 日本でもReact開発者が最も多く、人材確保しやすい
5. **モジュラー設計**: コンポーネントベースの設計がモジュール分割と相性が良い
6. **2024年の調査**: JavaScript開発者の57%がReactを使用（最大シェア）

## 影響

### ポジティブな影響
- 豊富な学習リソースとコミュニティサポート
- UIライブラリ（shadcn/ui等）の選択肢が多い
- TypeScriptによる型安全性でバグを早期発見
- 長期的なメンテナンスが期待できる

### ネガティブな影響
- 状態管理など追加の設計決定が必要
- Reactの更新に追従する必要がある
- 設計パターンの統一にドキュメント・レビューが必要

### リスクと軽減策
| リスク | 軽減策 |
|--------|--------|
| 設計のばらつき | SDDでパターンを明文化、コードレビュー徹底 |
| React大型アップデート | 安定版を使用、段階的なアップグレード計画 |
| バンドルサイズ肥大 | 定期的なバンドル分析、遅延読み込み活用 |
