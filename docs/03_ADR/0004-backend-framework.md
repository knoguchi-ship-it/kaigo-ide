# ADR-0004: バックエンドフレームワークの選定

**ステータス**: Accepted
**日付**: 2026-01-29
**決定者**: [アーキテクト名]

## コンテキスト

KaigoIDEのバックエンドAPIを開発するフレームワークを選定する必要がある。

要件：
- TypeScriptでの開発（フロントエンドと言語を統一）
- モジュラーアーキテクチャのサポート
- REST API開発に適した機能
- 長期的な保守性とコミュニティサポート
- 認証・認可の実装が容易

## 決定

**NestJS を バックエンドフレームワークとして採用する。**

## 選択肢

### 選択肢1: NestJS
- **概要**: Angularに影響を受けた、構造化されたNode.jsフレームワーク
- **メリット**:
  - モジュール・サービス・コントローラーの明確な構造
  - TypeScriptファースト
  - 依存性注入（DI）のサポート
  - デコレータによる宣言的な記述
  - OpenAPI/Swagger自動生成
  - 認証（Passport）の統合が容易
- **デメリット**:
  - 学習コストがやや高い
  - ボイラープレートがやや多い
  - オーバーキルになる可能性

### 選択肢2: Express.js
- **概要**: 最も広く使われているNode.jsフレームワーク
- **メリット**:
  - シンプルで学習コストが低い
  - 最大のエコシステム
  - 柔軟性が高い
- **デメリット**:
  - 構造が規定されていない
  - TypeScriptサポートは追加設定が必要
  - 大規模開発では自前で構造化が必要

### 選択肢3: Fastify
- **概要**: 高速性を重視したNode.jsフレームワーク
- **メリット**:
  - Expressより高速
  - スキーマベースのバリデーション
  - プラグインシステム
- **デメリット**:
  - Expressほどのエコシステムがない
  - 大規模開発の構造は自前で設計が必要

### 選択肢4: Hono
- **概要**: 軽量で高速なWebフレームワーク
- **メリット**:
  - 非常に軽量・高速
  - Edge環境対応
  - シンプルなAPI
- **デメリット**:
  - 比較的新しく実績が少ない
  - 大規模開発のパターンが確立されていない

## 決定理由

1. **モジュラー設計**: NestJSのモジュール構造がKaigoIDEのモジュール分割方針と合致
2. **TypeScriptファースト**: 追加設定なしでTypeScriptが使える
3. **構造化**: コントローラー・サービス・リポジトリの階層が明確
4. **DIコンテナ**: テスタビリティが高く、モックが容易
5. **認証統合**: Passportとの統合が標準でサポート
6. **OpenAPI**: デコレータからSwaggerドキュメントを自動生成

## 影響

### ポジティブな影響
- プロジェクト全体で一貫した構造が保てる
- 新メンバーが構造を理解しやすい
- DIによりテストが書きやすい
- APIドキュメントが自動生成される

### ネガティブな影響
- Expressより学習コストが高い
- 小規模な機能でもボイラープレートが必要
- フレームワークへの依存度が高い

### リスクと軽減策
| リスク | 軽減策 |
|--------|--------|
| 学習コスト | 公式チュートリアルの実施、ペアプログラミング |
| オーバーエンジニアリング | 必要最小限のモジュール構成から開始 |
| バージョンアップ | LTS版を使用、段階的なアップグレード |
