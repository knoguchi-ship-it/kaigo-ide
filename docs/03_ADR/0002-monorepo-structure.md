# ADR-0002: モノレポ構成の採用

**ステータス**: Accepted
**日付**: 2026-01-29
**決定者**: [アーキテクト名]

## コンテキスト

KaigoIDEは複数のモジュール（支援経過記録、担当者会議、ケアプラン等）で構成される。また、フロントエンドとバックエンドを同一チームで開発する。リポジトリ構成を決定する必要がある。

要件：
- モジュール間でコードを共有したい
- フロント・バックエンドの型定義を共有したい
- 独立したモジュール開発を可能にしたい
- CI/CDパイプラインを効率化したい

## 決定

**pnpm workspaces + Turborepo によるモノレポ構成を採用する。**

```
kaigoIDE/
├── apps/
│   └── web/              # フロントエンド
├── services/
│   └── api/              # バックエンドAPI
├── packages/
│   ├── ui/               # 共通UIコンポーネント
│   ├── types/            # 共通型定義
│   └── utils/            # 共通ユーティリティ
├── pnpm-workspace.yaml
└── turbo.json
```

## 選択肢

### 選択肢1: モノレポ（pnpm + Turborepo）
- **概要**: 単一リポジトリで複数パッケージを管理
- **メリット**:
  - コード共有が容易
  - 型定義の共有でフロント・バック間の整合性
  - 一括でのlint/test/build
  - 依存関係の一元管理
  - Turborepoによるキャッシュで高速ビルド
- **デメリット**:
  - 初期セットアップの学習コスト
  - リポジトリサイズが大きくなる
  - 細かい権限管理が難しい

### 選択肢2: ポリレポ（複数リポジトリ）
- **概要**: フロントエンド、バックエンド、共通パッケージを別リポジトリで管理
- **メリット**:
  - リポジトリが小さく見通しが良い
  - チーム別の権限管理が容易
  - 独立したデプロイ
- **デメリット**:
  - コード共有にnpm publish等が必要
  - 型定義の同期が煩雑
  - 変更時の整合性確保が難しい

### 選択肢3: モノレポ（Nx）
- **概要**: Nrwlが開発するモノレポツール
- **メリット**:
  - 強力なコード生成機能
  - 依存グラフの可視化
  - プラグインエコシステム
- **デメリット**:
  - 学習コストが高い
  - 設定が複雑
  - Turborepoより重い

### 選択肢4: モノレポ（Lerna）
- **概要**: 古くからあるモノレポツール
- **メリット**:
  - 実績が多い
  - シンプルな設計
- **デメリット**:
  - メンテナンスが停滞していた時期がある
  - Turborepoほど高速ではない

## 決定理由

1. **コード共有の容易さ**: フロント・バックエンドで型定義を共有でき、整合性を保てる
2. **開発効率**: pnpmの高速なインストールとTurborepoのキャッシュで効率的
3. **段階的開発**: モジュールを独立して追加していく開発スタイルに適合
4. **学習コスト**: NxよりTurborepoの方がシンプルで導入しやすい
5. **エコシステム**: pnpmは広く使われており、サポートも充実

## 影響

### ポジティブな影響
- `packages/types`で定義した型をフロント・バック両方で使用可能
- UIコンポーネントの再利用性向上
- 全体のlint/test/buildを一括実行可能
- TurborepoのリモートキャッシュでCI/CDが高速化

### ネガティブな影響
- 初期セットアップに時間が必要
- 新メンバーへのモノレポ運用の説明が必要
- 全体のビルドが複雑になる可能性

### リスクと軽減策
| リスク | 軽減策 |
|--------|--------|
| 設定の複雑化 | READMEとコントリビューションガイドを整備 |
| ビルド時間の増加 | Turborepoキャッシュ、影響範囲のみビルド |
| 依存関係の衝突 | pnpmの厳格な依存解決、定期的な更新 |
