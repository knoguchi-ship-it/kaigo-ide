# ADR-0003: 状態管理ライブラリの選定

**ステータス**: Accepted
**日付**: 2026-01-29
**決定者**: [アーキテクト名]

## コンテキスト

React単体では大規模なアプリケーションの状態管理が難しい。KaigoIDEでは以下の状態を管理する必要がある：

- **サーバー状態**: 利用者情報、記録データ等（APIから取得）
- **グローバルUI状態**: 認証情報、テーマ、サイドバー開閉等
- **ローカル状態**: フォーム入力、モーダル表示等

適切な状態管理戦略を決定する必要がある。

## 決定

**以下の組み合わせを採用する：**

| 状態タイプ | ライブラリ |
|-----------|-----------|
| サーバー状態 | TanStack Query (React Query) |
| グローバルUI状態 | Zustand |
| ローカル状態 | useState / useReducer |
| フォーム状態 | React Hook Form |

## 選択肢

### サーバー状態管理

#### 選択肢A1: TanStack Query
- **概要**: サーバー状態に特化したデータフェッチングライブラリ
- **メリット**:
  - キャッシュ、バックグラウンド更新、楽観的更新
  - ローディング/エラー状態の自動管理
  - DevToolsによるデバッグ
  - 軽量（サーバー状態のみ）
- **デメリット**:
  - サーバー状態以外は別途管理必要

#### 選択肢A2: SWR
- **概要**: Vercelが開発するデータフェッチングライブラリ
- **メリット**:
  - シンプルなAPI
  - Next.jsとの親和性
- **デメリット**:
  - TanStack Queryより機能が少ない
  - Mutation周りが弱い

#### 選択肢A3: Redux + RTK Query
- **概要**: Reduxエコシステムのデータフェッチング
- **メリット**:
  - Reduxと統合
  - 強力なキャッシュ機能
- **デメリット**:
  - Reduxの学習コスト
  - ボイラープレートが多い

### グローバルUI状態管理

#### 選択肢B1: Zustand
- **概要**: シンプルで軽量な状態管理ライブラリ
- **メリット**:
  - 学習コストが低い
  - ボイラープレートが少ない
  - TypeScriptサポートが良好
  - バンドルサイズが小さい（約1KB）
- **デメリット**:
  - 大規模アプリでの実績がReduxより少ない

#### 選択肢B2: Redux Toolkit
- **概要**: Redux公式の推奨ツールキット
- **メリット**:
  - 実績が豊富
  - DevToolsが充実
  - ミドルウェアが豊富
- **デメリット**:
  - 学習コストが高い
  - オーバーキルになりがち

#### 選択肢B3: Jotai
- **概要**: アトムベースの状態管理
- **メリット**:
  - Reactらしい設計
  - 細粒度の再レンダリング制御
- **デメリット**:
  - 新しめで実績が少ない
  - 設計パターンが独特

### フォーム状態管理

#### 選択肢C1: React Hook Form
- **概要**: パフォーマンス重視のフォームライブラリ
- **メリット**:
  - 非制御コンポーネントで高パフォーマンス
  - バリデーション統合（Zod, Yup）
  - バンドルサイズが小さい
- **デメリット**:
  - 非制御コンポーネントの概念に慣れが必要

#### 選択肢C2: Formik
- **概要**: 人気のフォームライブラリ
- **メリット**:
  - 制御コンポーネントで直感的
  - ドキュメントが充実
- **デメリット**:
  - パフォーマンスがReact Hook Formより劣る
  - バンドルサイズが大きい

## 決定理由

### TanStack Queryの採用理由
1. サーバー状態に特化した設計で、キャッシュ・同期が自動化される
2. ローディング/エラー/成功状態の管理が宣言的
3. 楽観的更新のサポートでUX向上
4. Reduxで同じことをするより遥かにシンプル

### Zustandの採用理由
1. 学習コストが非常に低く、すぐに使い始められる
2. ボイラープレートがほぼない
3. 必要な機能（永続化、DevTools）はミドルウェアで追加可能
4. KaigoIDEのグローバルUI状態は限定的なのでZustandで十分

### React Hook Formの採用理由
1. 介護記録入力など、フォームが多いアプリケーション
2. 非制御コンポーネントによる高パフォーマンス
3. Zodとの統合でスキーマベースのバリデーション

## 影響

### ポジティブな影響
- 各ライブラリが得意な領域に特化し、全体として効率的
- TanStack Queryによりデータ取得周りのコードが大幅削減
- Zustandによりグローバル状態のコードがシンプル
- 学習コストが低く、新メンバーが参加しやすい

### ネガティブな影響
- 複数のライブラリを組み合わせる必要がある
- ライブラリ間の連携パターンを理解する必要がある

### リスクと軽減策
| リスク | 軽減策 |
|--------|--------|
| ライブラリ間の責務の曖昧さ | SDDで明確な使い分けを文書化 |
| バージョンアップ | 定期的な依存関係更新、テストカバレッジ確保 |

## 使用例

```typescript
// サーバー状態（TanStack Query）
const { data, isLoading, error } = useCareRecords(clientId);

// グローバルUI状態（Zustand）
const { sidebarOpen, toggleSidebar } = useUIStore();

// フォーム状態（React Hook Form）
const form = useForm<CareRecordInput>({
  resolver: zodResolver(careRecordSchema),
});
```
