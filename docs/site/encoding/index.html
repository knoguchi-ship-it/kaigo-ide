文字化けは 2 種類に分けて扱う

表示だけ壊れているケースと、実ファイル自体が壊れているケースを分離して扱うことが、誤修復を防ぐための基本方針です。

結論

このリポジトリの標準文字コードは UTF-8 です。Windows PowerShell 5.1 の表示問題と、実ファイル破損は別問題として扱います。

表示時の文字化け

UTF-8 の正常ファイルでも、Windows PowerShell 5.1 と code page 932 の組み合わせでは文字化けして見えることがあります。

表示問題 実体は正常な場合がある

実ファイル内の文字化け

一部の API サービスでは、例外メッセージ文字列そのものが壊れて保存されています。こちらはコード修復が必要です。

実データ破損 個別修復が必要

確認済みの原因

  1. UTF-8 ファイルを CP932 前提で読むと文字化けする
  2. Windows PowerShell 5.1 は文字コード取り扱いで混乱を起こしやすい
  3. これまで `.editorconfig` と `.gitattributes` がなく、リポジトリとして文字コード方針が固定されていなかった

修復順序

  1. 文書で方針を固定する
  2. `.editorconfig` と `.gitattributes` を追加する
  3. 壊れた例外メッセージを個別修復する

  • care-plan.service.ts
  • client.service.ts
