Governance

開発をブレさせないための運用基準

このページは、プロジェクト・グランドルールを人が短時間で把握するための要約です。正本は Markdown です。

結論

KaigoIDE では、最新情報の Web 調査、根拠付き提案、明示承認、ドキュメント先行更新、誤解のない handover を必須とします。

Research First

設計、実装方針、セキュリティ、クラウド構成、API 設計は、実行日時点の最新情報を Web で確認してから着手します。

Proposal First

実装前に、根拠付きのベストプラクティス提案を出し、推奨案と代替案を整理します。

Docs First

コード変更より先に、要件、設計、運用、環境設定、handover の文書を更新します。

標準作業順序

  1. Web で最新情報を調査する
  2. 根拠付きでベストプラクティス提案を行う
  3. 明示的な承認を得る
  4. 関連ドキュメントを先に更新する
  5. 実装する
  6. 検証する
  7. 必要に応じて引き継ぎ文書を更新する

禁止事項

  • 記憶だけで設計判断する
  • 提案前に実装する
  • 承認前にコードを書き進める
  • ドキュメント更新を後回しにする
  • 曖昧な handover を残す
