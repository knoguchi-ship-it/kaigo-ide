# プロジェクト・グランドルール

最終更新日: 2026-03-20  
位置付け: 正本  
適用範囲: このリポジトリに関わる全ての開発者および AI エージェント

## 目的

この文書は、本リポジトリにおける最上位ルールを定める。

人間の開発者、Codex、Claude Code、Gemini その他の AI エージェントは、この文書を最優先で参照し、従わなければならない。

## 規範表現

この文書における「しなければならない」「してはならない」「推奨する」は、RFC 2119 および RFC 8174 の規範表現の考え方に準拠する。

参照:
- RFC 2119: https://www.rfc-editor.org/rfc/rfc2119
- RFC 8174: https://www.rfc-editor.org/rfc/rfc8174

## グランドルール

### GR-001 最新情報の Web 調査を必須とする

全ての実質的な作業に着手する前に、実行日時点での最新情報を Web 検索によって確認しなければならない。

対象には少なくとも以下を含む。
- 設計
- 実装方針
- セキュリティ判断
- クラウド構成
- API 設計
- 運用設計
- 依存ライブラリやサービス選定

記憶だけに依存して判断してはならない。
可能な限り、公式資料、一次情報、標準文書を優先しなければならない。

### GR-002 調査結果を根拠としたベストプラクティス提案を必須とする

GR-001 で取得した情報を根拠として、実装前にベストプラクティスの提案を行わなければならない。

提案には少なくとも以下を含める。
- 推奨案
- 推奨理由
- 必要に応じた代替案
- 根拠となる参照元

### GR-003 提案と承認なしに実装してはならない

提案を行う前に、勝手にコード実装へ進んではならない。
提案後、ユーザーまたは依頼者から明示的な承認を得るまでは、実装してはならない。

明示的な承認の例:
- 「それで進めてください」
- 「実装してください」
- 「その方針でお願いします」

### GR-004 ドキュメントドリブンで進めなければならない

コードを変更する前に、関連するドキュメントを先に更新するか、少なくとも更新案を提示しなければならない。

対象には少なくとも以下を含む。
- 要件
- 設計
- 運用
- 引き継ぎ
- 環境設定
- デプロイ手順

コードはドキュメントに従って変更されるべきであり、ドキュメントより先行してはならない。

### GR-005 引き継ぎは誤解なく完了できる品質で残さなければならない

引き継ぎの指示があった場合、次の開発者または AI エージェントが誤解なく継続できる状態を必ず残さなければならない。

引き継ぎには少なくとも以下を含める。
- 現在の開発状況
- 完了したこと
- 未完了のこと
- 次に着手すべきこと
- 目指すべき状態
- 判断理由
- 関連コードおよび関連ドキュメント
- リスクと未解決事項

### GR-006 グローバルスタンダードを満たさなければならない

本プロジェクトにおける設計、実装、運用、セキュリティ、レビュー、文書品質は、常にグローバルスタンダードの観点で判断しなければならない。

優先して参照すべきもの:
- 公式ベンダードキュメント
- IETF、W3C、ISO、CNCF、OpenSSF、OWASP 等の標準・ガイド
- 公式フレームワークドキュメント
- 一次情報

### GR-007 常に今より改善することを目指さなければならない

全ての変更は、可能な限り現在より良い状態を目指さなければならない。

少なくとも以下の改善を常に意識する。
- 明確さ
- 安全性
- 保守性
- 検証可能性
- 引き継ぎやすさ
- 運用しやすさ

やむを得ず改善できない場合、または一時的に妥協する場合は、その理由を明示しなければならない。

## 標準作業順序

特別な指示がない限り、全ての作業は以下の順序で進めなければならない。

1. Web で最新情報を調査する
2. 根拠付きでベストプラクティス提案を行う
3. 明示的な承認を得る
4. 関連ドキュメントを先に更新する
5. 実装する
6. 検証する
7. 必要に応じて引き継ぎ文書を更新する

## 情報源の優先順位

参照する情報源は、原則として次の順序で優先しなければならない。

1. 標準化団体、RFC
2. 公式クラウド、公式フレームワーク、公式製品ドキュメント
3. 公式セキュリティガイド
4. 一次情報
5. 一次情報で不足する場合に限った信頼できる二次情報

一次情報が存在する場合、ブログ、掲示板、AI 生成要約を権威情報として扱ってはならない。
